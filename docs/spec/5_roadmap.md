## 2. 本ドキュメントの位置づけ

本ドキュメントは、ゴール達成型タスク管理アプリの
**中長期的な進化計画を定義する公式ロードマップ**である。

ただし本ロードマップは、以下の前提条件に厳密に従う。

> **Phase2 以降のすべての機能は、
> 「構造は強制、思考はユーザー」という思想を損なわないことを前提とする。**

思想と矛盾する機能は、ユーザーニーズや技術的可能性があっても採用しない。

---

## 3. ロードマップ設計方針

### 3.1 基本方針

- MVP（Phase1）で提供する価値を最優先する
- 機能追加は「便利さ」ではなく「目的達成への寄与度」で判断する
- 思考を代行・短絡化する機能は追加しない

### 3.2 フェーズ定義

| フェーズ | 目的                       |
| -------- | -------------------------- |
| Phase1   | 核となる体験の完成（MVP）  |
| Phase1.5 | MVP改善・品質向上          |
| Phase2   | 利用継続・データ保全の強化 |
| Phase3   | 補助的価値の追加（慎重）   |

---

## 4. Phase1（MVP）

### 4.1 フェーズ目的

- 「目的 → 計画 → 行動」が自然に回る体験を完成させる
- 迷わず使えるUI/UXを提供する

### 4.2 提供機能（確定）

| No  | 機能               | 説明                             |
| --- | ------------------ | -------------------------------- |
| 1   | ゴール管理         | 複数ゴール作成・編集・削除       |
| 2   | マイルストーン管理 | ゴール配下でのみ作成             |
| 3   | タスク管理         | MS配下、3状態（Todo/Doing/Done） |
| 4   | 進捗自動計算       | タスク→MS→ゴール                 |
| 5   | ゴール一覧         | 期限・進捗の可視化               |
| 6   | ゴール詳細         | 計画と行動の中心画面             |
| 7   | リストビュー       | デフォルト表示                   |
| 8   | ピラミッドビュー   | 階層構造の理解支援               |
| 9   | カレンダービュー   | 期限ベース把握                   |
| 10  | 今日のタスク       | 行動集中ビュー                   |
| 11  | オンボーディング   | スキップ不可、ゴール作成必須     |
| 12  | ローカル保存       | オフライン前提                   |

---

## 5. Phase1.5（改善フェーズ）

### 5.1 フェーズ目的

- MVPの体験を壊さずに使い勝手を改善する
- 初期ユーザーの不満点を解消する

### 5.2 提供候補機能

| No  | 機能               | 理由                   |
| --- | ------------------ | ---------------------- |
| 1   | ゴール検索         | ゴール数増加時の利便性 |
| 2   | UI微調整           | 可読性・操作性改善     |
| 3   | パフォーマンス改善 | スクロール・描画最適化 |
| 4   | 軽微なUX改善       | 文言・導線の改善       |

※ 思想・構造には影響を与えない範囲に限定する

---

## 6. Phase2（継続利用・保全強化）

### 6.1 フェーズ目的

- データ喪失リスクの低減
- 端末変更時の継続利用を可能にする

### 6.2 提供候補機能

| No  | 機能               | 方針                    |
| --- | ------------------ | ----------------------- |
| 1   | クラウド同期       | ローカルファースト設計  |
| 2   | アカウント機能     | 同期用途に限定          |
| 3   | データバックアップ | エクスポート/インポート |

※ 通知機能は本フェーズでも必須としない

---

## 7. Phase3（補助的価値の追加）

### 7.1 フェーズ目的

- ゴール達成体験を妨げない範囲での付加価値提供

### 7.2 検討対象（慎重）

| 機能             | 採用可否の考え方             |
| ---------------- | ---------------------------- |
| 振り返りレポート | 行動の内省を促すなら可       |
| ウィジェット     | 行動を促すなら可             |
| 軽微な通知       | 任意・依存させない設計なら可 |

### 7.3 明示的に採用しないもの

- ゴール・計画の自動分解
- AIによる思考代行
- 強制的なテンプレート
- SNS共有・ランキング

---

## 8. 実装しない機能（思想的に除外）

以下は、将来にわたって実装対象外とする。

- チームタスク管理
- 業務KPI管理
- 習慣トラッカー
- 自由構造ノート

---

## 9. リリース・デプロイプロセス

### 9.1 MVP リリーススケジュール（Phase1：2026年6月リリース）

**開発期間**：2026年3月～6月初旬（約3ヶ月）

**リリース前準備期間**：2026年6月中旬～6月下旬（2週間）

以下の準備を実施する：

- **App Store / Play Store 申請準備**（6月中旬から開始）
  - プライバシーポリシー作成・掲載
  - 利用規約作成・掲載
  - アプリ説明文・スクリーンショット用意
  - 年齢レーティング登録
  - カテゴリ・キーワード設定
- **リリース申請**（6月下旬）

### 9.2 ベータテストについて

- **MVP段階ではベータテスト（TestFlight / Google Play Beta）を実施しない**
- 理由：ユーザーベースが構築できていない段階での限定テストより、段階的なリリース後の実ユーザーフィードバックを優先

### 9.3 クラッシュログについて

- **Firebase Crashlytics 等のクラッシュ分析は MVP には含めない**
- 理由：初期段階では QA により十分な品質確保が可能
- Phase2で検討予定

---

## 10. ロードマップの運用ルール

- 各フェーズ開始前に、必ずコンセプト定義書との整合性を確認する
- 新機能提案時は「この機能は思考を奪っていないか？」を必ず検討する
- 思想と衝突する場合、ユーザーニーズより思想を優先する

---

## 11. まとめ

本ロードマップは、
**機能拡張の計画書ではなく、思想を守りながら成長するための指針**である。

プロダクトは、便利さではなく
「ユーザーが目的に向かって前進できているか」を基準に進化させる。
