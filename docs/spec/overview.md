## 2. 全体像サマリー

### 2.1 プロダクトコンセプト

> **「目的達成のために、今日やることが分かるアプリ」**

### 2.2 解決する課題

既存タスク管理アプリの多くは自由度が高く、ユーザー自身が構造を設計する必要があります。
本アプリは、以下の課題を解決します：

- 何のためのタスクか分からなくなる
- ゴールから逆算した行動計画が立てられない
- 継続利用されにくい

**本質的課題**：「目的 → 計画 → 行動」への分解を人力で行うのは難しい

### 2.3 想定ユーザー

**「目的を達成したいと思うすべての人」**

特に以下のユーザーに効果的：

- 受験生（学習計画の不安解消）
- 就活生（キャリア構築）
- ダイエット実行者（目標達成への継続）
- 趣味実行者（目標達成への意欲）

### 2.4 差別化ポイント

**「方向性3：シンプルさと強制力による継続性」**

| ポイント         | 他アプリ | このアプリ                        |
| ---------------- | -------- | --------------------------------- |
| **設計の自由度** | 高い     | 強制（ゴール → MS → タスク のみ） |
| **ユーザー負荷** | 高い     | 低い                              |
| **UI複雑さ**     | 複雑     | シンプル                          |
| **習慣化難度**   | 難しい   | 簡単                              |

### 2.5 コア思想

- **構造は強制、分解はユーザー**：アプリが3階層構造を強制するが、具体的な分解はユーザーが考える
- **思考負荷を極限まで減らす**：「どう使うか」について考えさせない
- **ゴール達成体験を最優先**：すべての設計判断は達成可能性を基準に

## 3. 主要な設計決定

### 3.1 ビジネス目標

| 目標             | 値                  | 時期         |
| ---------------- | ------------------- | ------------ |
| 月間売上         | 100万円             | 1年以内      |
| 月間ダウンロード | 1,000DL             | 3ヶ月以内    |
| 価格             | 1,000円（買い切り） | 初回リリース |

### 3.2 技術選定

| 項目           | 選択                             | 理由                                             |
| -------------- | -------------------------------- | ------------------------------------------------ |
| フレームワーク | **Flutter**                      | クロスプラットフォーム、高速開発                 |
| 状態管理       | **Riverpod**                     | 型安全、テスト性が高い                           |
| データ保存     | **Hive**                         | シンプル、高速、将来のクラウド同期に対応しやすい |
| アーキテクチャ | **クリーンアーキテクチャ + DDD** | テスタビリティ、スケーラビリティ                 |

### 3.3 データ構造

```
ゴール
├── マイルストーン（中間目標）
│   └── タスク（日々の行動）
└── マイルストーン
    └── タスク
```

**進捗は自動計算**：

- タスク：状態ベース（0% / 50% / 100%）
- マイルストーン：配下タスク平均
- ゴール：配下マイルストーン平均

### 3.4 UI/UX 方針

| 要素               | 方針                                |
| ------------------ | ----------------------------------- |
| **ナビゲーション** | ボトムタブ3つ（ホーム・今日・設定） |
| **情報密度**       | 1画面1役割で抑制                    |
| **迷わなさ**       | ゴール → MS → タスク の一本道       |

## 4. MVP（最小限の製品）仕様

### 4.1 実装される機能

| 機能                   | 内容                                         | 優先度 |
| ---------------------- | -------------------------------------------- | ------ |
| **ゴール管理**         | CRUD、複数作成可能                           | MUST   |
| **マイルストーン管理** | ゴール配下でのみ作成                         | MUST   |
| **タスク管理**         | MS配下でのみ作成、3状態                      | MUST   |
| **進捗管理**           | 自動計算による可視化                         | MUST   |
| **表示機能**           | リスト・ピラミッド・カレンダー・今日のタスク | MUST   |
| **ローカル保存**       | Hive によるオフライン対応                    | MUST   |
| **オンボーディング**   | 初回ユーザーガイド                           | MUST   |

### 4.2 非実装機能（Phase 2以降）

| 機能         | 計画時期           | 理由                   |
| ------------ | ------------------ | ---------------------- |
| クラウド同期 | Phase 2（1年以内） | インフラ構築のコスト   |
| 通知機能     | Phase 2            | MVP では不要           |
| AI提案       | Phase 3（2年以内） | ユーザーの主体性を重視 |
| チーム機能   | Phase 4（3年以上） | 個人利用に特化         |

## 8. 用語集

| 用語                       | 定義                                                 |
| -------------------------- | ---------------------------------------------------- |
| **ゴール**                 | ユーザーが達成したい最終的な目的（例：志望校合格）   |
| **マイルストーン（MS）**   | ゴール達成のための中間目標（例：共通テスト対策完了） |
| **タスク**                 | MS達成のための具体的な行動（例：英語の過去問を解く） |
| **進捗**                   | ゴール/MS/タスク の達成度（%で表示）                 |
| **MVP**                    | 最小限の機能を備えた製品版（Minimum Viable Product） |
| **クリーンアーキテクチャ** | 依存関係の方向性に基づくアーキテクチャパターン       |
| **Riverpod**               | Dart/Flutter の状態管理フレームワーク                |
| **Hive**                   | Dart 用の高速ローカルデータベース                    |
